# NLP Sentiment Classification 

Three-class sentiment classification (Negative `-1`, Neutral `0`, Positive `+1`)
over a merged corpus of Amazon product reviews (Ni et al. 2019) and SemEval-2017
Task 4 tweets (Nakov et al. 2016). The deliverable is a reproducible pipeline
plus a pickled model served behind an API and scored on a private test set
(macro-F1).

VU Amsterdam — Data Science Project. Built and run on the VU Compute Hub
(JupyterHub, `Data Science Project` kernel, Python 3.11).

## Approach

The work is organised into three independent modelling routes. Each route loads
the pre-split data on its own and applies its own preprocessing, so a decision in
one route never leaks into another.

- **Route A — TF-IDF baseline** *(implemented)*
  TF-IDF encoder, chi-squared feature selection (`SelectKBest`), classical
  classifiers (Naïve Bayes, Logistic Regression, Linear SVM). Framed as a
  well-documented baseline for Route C to beat.
- **Route B — Static embeddings** *(planned)*
  Word2Vec / GloVe / fastText averaged document vectors + classical classifiers.
  Uses lighter tokenisation that preserves word forms.
- **Route C — Contextual embeddings** *(planned)*
  BERTweet (Nguyen et al. 2020). The intended path to top performance; starts
  from raw text using BERTweet's own normaliser/tokeniser.

## Data

Not redistributed in this repo.

- **Source database** `nlp-data.db` lives only on the VU Compute Hub at
  `/local/DSPT/data/nlp-data.db` (read-only). `EDA.ipynb` queries it directly.
- **`splits/data_split.csv`** is a derived artifact (columns: `clean_eda`,
  `sentiment`, `Type`, `split`). It is git-ignored and regenerated by running
  `EDA.ipynb`. Every route reads it independently.

## Reproduce

1. Launch on the VU Compute Hub with the `Data Science Project` kernel, or set up
   a local Python 3.11 env: `pip install -r requirements.txt`.
2. First run only — NLTK data:
   ```python
   import nltk
   nltk.download("stopwords"); nltk.download("wordnet"); nltk.download("omw-1.4")
   ```
3. Run `EDA.ipynb` end to end → writes `splits/data_split.csv`.
4. Run `route_a.ipynb` → trains, evaluates on the held-out test split, and pickles
   the deployment dict to `route_a.model`.

## Current status

| Route | State        | Notes |
|-------|--------------|-------|
| A     | Completed  | TD-IDF + Logistic Regression best; 0.7 test macro-F1 |
| B     | Completed  | Sentence-transformer Embeddings + Logistic Regression best; 0.64 test macro F1 |
| C     | Completed  | BERTweet-large best; 0.76 test macro f1 |

Route A notes: neutral class is the main drag (F1 ≈ 0.52) and tweets underperform
reviews; `k` past ~19k features barely moves the needle. Native tokenisation is
preferred over lemmatisation (delta within noise). The deployed encoder is a
plain `TfidfVectorizer` (no notebook-defined tokenizer) so the pickle loads
cleanly in a separate API process; inference uses `transform`, never
`fit_transform`.

## Deployment contract

Per the case manual, the packaged `.model` is a dict with two keys:

```python
deployment = {"vectorizer": <fitted vectorizer>, "classifier": <trained model>}
```

The API loads only this file — no other notebook objects are available at load
time.
